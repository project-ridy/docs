# Ridy — B2B 기능 기획

## 1. 목적

Ridy의 B2B 기능은 기업이 사내 카풀을 안전하게 도입하고 운영하도록 돕는 관리자·구독·리포팅 기능 묶음이다.

기존 MVP가 사원 간 카풀 매칭, 채팅, 정산을 검증한다면 B2B 기능은 다음 질문에 답한다.

- 기업 관리자는 사내 카풀 참여자를 어떻게 초대하고 관리하는가?
- 회사 요금제에 따라 정산 수수료 부담은 어떻게 달라지는가?
- 기업은 카풀 이용 현황과 CO₂ 절감 효과를 어떻게 확인하는가?
- 다중 기업 환경에서 데이터 격리는 어떻게 유지되는가?

## 3. B2B 사용자와 Job-to-be-Done

### 3.1 기업 관리자

- **상황**: 사내 출퇴근 교통비 부담과 ESG 실적 요구가 커진다.
- **목표**: 검증된 사원만 참여하는 카풀 프로그램을 운영하고 성과를 확인한다.
- **성공 기준**: 초대 코드로 구성원을 통제하고, 이용 통계·정산·CO₂ 절감량을 한 화면에서 확인한다.

### 3.2 사원

- **상황**: 회사가 Ridy를 도입했지만 외부인 노출이나 정산 부담이 걱정된다.
- **목표**: 같은 회사 구성원과만 카풀하고, 회사 플랜에 맞는 비용 분담을 적용받는다.
- **성공 기준**: 가입·매칭·채팅·정산 전 과정에서 회사 경계와 플랜 정책이 일관되게 적용된다.

### 3.3 Ridy 운영자

- **상황**: 여러 기업 고객을 온보딩하고 플랜별 계약을 관리해야 한다.
- **목표**: FREE/PRO/ENTERPRISE 플랜을 기준으로 기능 접근과 비용 부담을 구분한다.
- **성공 기준**: 기업별 설정이 데이터 격리, 정산, 대시보드 노출 범위에 반영된다.

## 4. 범위

### 4.1 포함 범위

1. **기업 플랜 정의**
   - `FREE`: 기본 매칭/채팅/정산, 사원 전액 부담
   - `PRO`: 관리자 대시보드, ESG 리포트, 회사 50% 부담
   - `ENTERPRISE`: 맞춤 도메인, 전담 지원, 회사 100% 부담
2. **관리자 운영 기능**
   - 초대 코드 생성·비활성화
   - 사원 목록/가입 현황 확인
   - 회사 단위 이용 통계 확인
   - 회사 정산 통계 확인
3. **ESG 리포팅**
   - CO₂ 절감량, 카풀 운행 수, 참여자 수 표시
   - 기업 관리자용 리포트와 사원용 친환경 임팩트 대시보드의 지표 정의 일관성 유지
4. **B2B 구독 결제 기획**
   - 플랜별 기능 차등
   - 플랜별 정산 수수료 부담 주체
   - 오픈 베타 이전 수동 계약/청구와 자동 구독 결제의 경계 정의
5. **다중 기업 운영 원칙**
   - 모든 조회·변경은 요청자의 `companyId`로 자동 스코프 처리
   - 타 기업 사원, 운행, 채팅, 정산, 리포트 접근 금지

### 4.2 제외 범위

- 실제 세금계산서 발행, 회계 시스템 연동
- 관리자 ESG 리포트 PDF 생성
- CRM, HRIS, SSO/SAML 연동
- 회사별 커스텀 도메인의 DNS/인프라 자동화
- ENTERPRISE 전담 지원 운영 도구

위 항목은 B2B 기능의 후속 확장 후보이며, 현재 Phase 4~5 문서 범위에서는 구현 스펙으로 확정하지 않는다.

## 5. 플랜 정책

| 플랜 | 대상 | 사원 수 기준 | 정산 수수료 부담 | 제공 기능 | 오픈 베타 기준 |
|---|---|---:|---|---|---|
| FREE | 파일럿/소규모 팀 | 최대 50명 | 사원 전액 부담 | 초대 코드, 매칭, 채팅, 정산 | 기본 제공 |
| PRO | 일반 기업 고객 | 최대 200명 | 회사 50% + 사원 50% | FREE + 관리자 대시보드 + ESG 리포트 | 우선 판매 플랜 |
| ENTERPRISE | 대기업/맞춤 계약 | 무제한 | 회사 100% | PRO + 맞춤 도메인 + 전담 지원 | 수동 계약 기반 |

플랜 수치와 기능 경계는 [DATABASE.md](../architecture/DATABASE.md)의 `Plan` enum 및 [PAYMENT.md](../api/PAYMENT.md)의 정산 정책과 일치해야 한다.

## 6. 관리자 대시보드 정보 구조

| 영역 | 주요 정보 | 문서 기준 | 플랜 |
|---|---|---|---|
| 초대 코드 | 코드, 활성 여부, 사용 횟수, 만료일 | [AUTH.md](../api/AUTH.md) | FREE 이상 |
| 사원 관리 | 이름, 사번, 역할, 가입일, 운행/탑승 수 | [DATABASE.md](../architecture/DATABASE.md) | PRO 이상 |
| 이용 통계 | 월간 카풀 수, 활성 사용자, 인기 경로 | [ARCHITECTURE.md](../architecture/ARCHITECTURE.md) | PRO 이상 |
| 정산 통계 | 총 결제/수령/회사 부담/사원 부담 | [PAYMENT.md](../api/PAYMENT.md) | PRO 이상 |
| ESG 리포트 | CO₂ 절감량, 누적 절감량, 참여율 | [PAYMENT.md](../api/PAYMENT.md) | PRO 이상 |
| 계약/플랜 | 현재 플랜, 사원 한도, 업그레이드 CTA | [API.md](../api/API.md) | 전체 |

관리자 대시보드는 사원 앱과 같은 Next.js 클라이언트를 사용할 수 있으나, 모든 데이터 접근은 관리자 권한과 `companyId` 스코프 검증을 통과해야 한다.

## 7. B2B 구독과 정산 정책

### 7.1 사원 정산 정책

- 회사 플랜별 수수료 계산은 [PAYMENT.md](../api/PAYMENT.md)의 `calculateFare` 정책을 따른다.
- `FREE`: 회사 부담 없음, 사원이 플랫폼 수수료 전액 부담
- `PRO`: 회사와 사원이 플랫폼 수수료를 절반씩 부담
- `ENTERPRISE`: 회사가 플랫폼 수수료 전액 부담, 사원 수수료는 0원

### 7.2 기업 구독 정책

- Phase 5 오픈 베타의 B2B 구독 결제는 플랜 선택과 청구 상태 관리가 핵심이다.
- 자동 구독 결제 API가 문서화되기 전까지 기업 구독 결제 구현은 `BLOCKED`로 간주한다.
- 단, 사원 정산에서 사용하는 `Company.plan` 기반 수수료 정책은 이미 [API.md](../api/API.md), [PAYMENT.md](../api/PAYMENT.md), [DATABASE.md](../architecture/DATABASE.md)에 정의되어 있다.

## 8. ESG 리포트 정책

| 지표 | 설명 | 노출 대상 | 기준 문서 |
|---|---|---|---|
| 카풀 운행 수 | 완료된 카풀 운행 수 | 관리자, 사원 | [PAYMENT.md](../api/PAYMENT.md) |
| 참여자 수 | 회사 내 카풀 참여 사용자 수 | 관리자 | [DATABASE.md](../architecture/DATABASE.md) |
| CO₂ 절감량 | 카풀로 줄인 예상 배출량 | 관리자, 사원 | [PAYMENT.md](../api/PAYMENT.md) |
| 누적 절감량 | 기간 누적 CO₂ 절감량 | 관리자, 사원 | [SCREENS.md](../design/SCREENS.md) |

CO₂ 산식은 API/구현 계획서에서 확정된 계산식을 따른다. 이 문서는 B2B 제품 범위와 노출 정책만 정의하며, 신규 산식을 임의로 추가하지 않는다.

| B2B-X01 | X | 초대 코드가 만료 또는 비활성화된 상태다 | 신규 가입은 거절되고 기존 사원 데이터는 유지된다 | [AUTH.md](../api/AUTH.md) |
| B2B-X02 | X | 회사가 FREE에서 PRO로 업그레이드한다 | 이후 정산부터 PRO 수수료 정책과 대시보드 접근이 적용된다 | [PAYMENT.md](../api/PAYMENT.md) |
| B2B-X03 | X | 다중 기업 데이터가 동시에 집계된다 | 리포트는 회사별로 분리되어 노출된다 | [ARCHITECTURE.md](../architecture/ARCHITECTURE.md) |

## 10. 구현 전 BLOCKED 조건

다음 작업은 별도 API/DB/디자인/구현 계획서가 생기기 전까지 구현하면 안 된다.

| 작업 | BLOCKED 사유 | 해제 조건 |
|---|---|---|
| 기업 구독 자동 결제 | GraphQL API, 결제 상태 모델, 웹훅 정책 미정 | `docs/api/`와 `docs/architecture/DATABASE.md`에 구독 계약 추가 |
| 관리자 ESG PDF | 산출물 형식, 다운로드 API, 저장 정책 미정 | 디자인/백엔드 구현 계획서 작성 |
| SSO/SAML | 인증 공급자, 도메인 검증, 계정 연결 정책 미정 | 인증 API 문서 확장 |
| 맞춤 도메인 자동화 | DNS/인프라/테넌트 라우팅 정책 미정 | 아키텍처 문서 확장 |

## 11. Phase별 실행 순서

| Phase | 목표 | 선행 조건 | 완료 기준 |
|---|---|---|---|
| Phase 4 | 1개 기업 파일럿 운영 | 초대 코드, 매칭, 채팅, 정산 MVP | 관리자 초대 코드/기본 통계 확인 가능 |
| Phase 5-1 | 다중 기업 오픈 베타 | 회사 단위 데이터 격리 검증 | 타 기업 데이터 접근 차단 케이스 통과 |
| Phase 5-2 | PRO 플랜 판매 | 관리자 대시보드와 ESG 리포트 | PRO 기능 접근/정산 정책 적용 |
| Phase 5-3 | ENTERPRISE 수동 계약 | 대기업 파일럿 요구사항 합의 | ENTERPRISE 수수료 정책과 운영 지원 범위 문서화 |

## 12. 완료 기준

- B2B 제품 범위가 `FREE/PRO/ENTERPRISE` 플랜 기준으로 정리되어 있다.
- 관리자 대시보드, ESG 리포트, B2B 구독 결제의 포함/제외 범위가 분리되어 있다.
- API, DB, 아키텍처, 디자인 문서와의 교차 참조가 명시되어 있다.
- 문서화되지 않은 API/DB/디자인 동작은 `BLOCKED` 조건으로 분리되어 있다.
