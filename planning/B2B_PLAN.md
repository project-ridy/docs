# Ridy — 후속 B2B 확장 기획

## 1. 목적

Ridy의 초기 모델은 회사 단위 계약이 아니라 **가입 코드 입력 → 회사 이메일 인증코드 → 비밀번호 설정 → 같은 회사 이메일 도메인끼리만 카풀**이다. 이 문서는 초기 도메인 기반 폐쇄형 커뮤니티가 검증된 뒤 확장할 B2B 계약·관리자·구독·리포팅 기능을 정리한다.

기존 MVP가 회사 이메일 도메인 기반 가입, 매칭, 채팅, 정산을 검증한다면 후속 B2B 기능은 다음 질문에 답한다.

- 도메인 운영자 또는 후속 기업 관리자는 사내 카풀 참여자를 어떻게 초대하고 관리하는가?
- 회사 단위 계약이 도입될 경우 요금제에 따라 정산 수수료 부담은 어떻게 달라지는가?
- 기업은 카풀 이용 현황과 CO₂ 절감 효과를 어떻게 확인하는가?
- 다중 기업 환경에서 데이터 격리는 어떻게 유지되는가?

## 3. B2B 사용자와 Job-to-be-Done

### 3.1 도메인 운영자

- **상황**: 사내 구성원끼리 안전하게 카풀을 시작하고 싶지만 회사 단위 계약까지는 이르다.
- **목표**: 가입 코드와 회사 이메일 도메인으로 검증된 구성원만 참여시키고 기본 현황을 확인한다.
- **성공 기준**: 다른 이메일 도메인의 가입·매칭이 차단되고, 같은 도메인 구성원끼리만 매칭된다.

### 3.2 사원

- **상황**: 사내 동료와만 카풀하고 싶지만 외부인 노출이나 신원 확인이 걱정된다.
- **목표**: 같은 회사 이메일 도메인 구성원과만 카풀하고, 개인이 이해 가능한 정산 금액을 확인한다.
- **성공 기준**: 가입·매칭·채팅·정산 전 과정에서 회사 이메일 도메인 경계가 일관되게 적용된다.

### 3.3 Ridy 운영자

- **상황**: 여러 회사 이메일 도메인의 클로즈드 커뮤니티를 운영하고, 이후 계약형 고객으로 확장할지 판단해야 한다.
- **목표**: 도메인별 가입·매칭 격리를 안정적으로 운영하고 B2B 전환 조건을 문서화한다.
- **성공 기준**: 초기 모델에서는 도메인 기반 격리만 구현하고, 회사 계약·플랜·회사 부담 정산은 별도 스펙 없이는 시작하지 않는다.

## 4. 범위

### 4.1 포함 범위

1. **초기 도메인 운영 정의**
   - 가입 코드 검증
   - 회사 이메일 도메인 검증
- 회사 이메일 인증코드 검증 및 `Company.domain` 도메인 일치 확인
2. **기본 운영 기능**
   - 가입 코드 생성·비활성화
   - 구성원 목록/가입 현황 확인
   - 회사 이메일 도메인 단위 기본 이용 통계 확인
3. **친환경 임팩트 표시**
   - CO₂ 절감량, 카풀 운행 수, 참여자 수 표시
   - 사원용 친환경 임팩트 대시보드와 지표 정의 일관성 유지
4. **후속 B2B 확장 검토**
   - 플랜별 기능 차등
   - 플랜별 정산 수수료 부담 주체
   - 수동 계약/청구와 자동 구독 결제의 경계 정의
5. **다중 도메인 운영 원칙**
   - 모든 조회·변경은 요청자의 `companyId`로 자동 스코프 처리
   - 타 기업 사원, 운행, 채팅, 정산, 리포트 접근 금지

### 4.2 제외 범위

- 회사 단위 계약, 영업, 청구, 세금계산서 발행, 회계 시스템 연동
- 관리자 ESG 리포트 PDF 생성
- CRM, HRIS, SSO/SAML 연동
- 회사별 커스텀 웹 도메인의 DNS/인프라 자동화
- PRO/ENTERPRISE 플랜 판매, 회사 부담 정산, 전담 지원 운영 도구

위 항목은 B2B 기능의 후속 확장 후보이며, 현재 MVP/클로즈드 베타 문서 범위에서는 구현 스펙으로 확정하지 않는다.

## 5. 플랜 정책 — 후속 확장 / 현재 구현 BLOCKED

초기 모델은 `FREE/PRO/ENTERPRISE`를 과금·기능 차등 기준으로 사용하지 않는다. 모든 MVP 사용자는 가입 코드와 회사 이메일 도메인 검증을 통과한 동일한 폐쇄형 커뮤니티 사용자로 취급한다. 아래 표는 후속 B2B 확장 검토용이며, API/DB/결제 스펙과 구현 계획서가 갱신되기 전까지 구현하면 안 된다.

| 플랜 | 대상 | 사원 수 기준 | 정산 수수료 부담 | 제공 기능 | 오픈 베타 기준 |
|---|---|---:|---|---|---|
| FREE | 후속 파일럿/소규모 팀 | 미정 | 미정 | 가입 코드, 매칭, 채팅, 정산 | 검토용 |
| PRO | 후속 일반 기업 고객 | 미정 | 미정 | FREE + 후속 관리자 대시보드 + ESG 리포트 | BLOCKED |
| ENTERPRISE | 후속 대기업/맞춤 계약 | 미정 | 미정 | PRO + 맞춤 웹 도메인 + 전담 지원 | BLOCKED |

플랜 수치와 기능 경계는 후속 API/DB/결제 스펙이 확정될 때 [DATABASE.md](../architecture/DATABASE.md)의 `Plan` enum 및 [PAYMENT.md](../api/PAYMENT.md)의 정산 정책과 동기화해야 한다.

도메인 운영 화면은 사원 앱과 같은 Next.js 클라이언트를 사용할 수 있으나, 모든 데이터 접근은 운영 권한과 `companyId`/`domain` 스코프 검증을 통과해야 한다.

## 7. B2B 구독과 정산 정책 — 현재 구현 BLOCKED

### 7.1 사원 정산 정책

- MVP 수수료 계산은 [PAYMENT.md](../api/PAYMENT.md)의 `calculateFare` 정책을 따른다.
- MVP에서는 회사 부담 수수료가 없고 탑승자가 플랫폼 수수료를 부담한다.
- `FREE/PRO/ENTERPRISE`에 따른 회사 부담 정산은 별도 API/DB/결제 스펙과 구현 계획서가 생기기 전까지 `BLOCKED`다.

### 7.2 기업 구독 정책

- Phase 5 이후 B2B 구독 결제는 플랜 선택과 청구 상태 관리가 핵심이다.
- 자동 구독 결제 API가 문서화되기 전까지 기업 구독 결제 구현은 `BLOCKED`로 간주한다.
- `Company.plan` 기반 수수료 정책은 현재 MVP 구현 근거가 아니며, 후속 확장 시 [API.md](../api/API.md), [PAYMENT.md](../api/PAYMENT.md), [DATABASE.md](../architecture/DATABASE.md)을 함께 갱신해야 한다.

## 8. ESG 리포트 정책

| 지표 | 설명 | 노출 대상 | 기준 문서 |
|---|---|---|---|
| 카풀 운행 수 | 완료된 카풀 운행 수 | 관리자, 사원 | [PAYMENT.md](../api/PAYMENT.md) |
| 참여자 수 | 같은 회사 이메일 도메인 내 카풀 참여 사용자 수 | 운영자 | [DATABASE.md](../architecture/DATABASE.md) |
| CO₂ 절감량 | 카풀로 줄인 예상 배출량 | 운영자, 사원 | [PAYMENT.md](../api/PAYMENT.md) |
| 누적 절감량 | 기간 누적 CO₂ 절감량 | 운영자, 사원 | [SCREENS.md](../design/SCREENS.md) |

CO₂ 산식은 API/구현 계획서에서 확정된 계산식을 따른다. 이 문서는 B2B 제품 범위와 노출 정책만 정의하며, 신규 산식을 임의로 추가하지 않는다.

## 10. 구현 전 BLOCKED 조건

다음 작업은 별도 API/DB/디자인/구현 계획서가 생기기 전까지 구현하면 안 된다.

| 작업 | BLOCKED 사유 | 해제 조건 |
|---|---|---|
| 기업 구독 자동 결제 | GraphQL API, 결제 상태 모델, 웹훅 정책 미정 | `docs/api/`와 `docs/architecture/DATABASE.md`에 구독 계약 추가 |
| 회사 부담 정산 | 플랜별 부담 비율, 청구 주체, 정산 회계 정책 미정 | `PAYMENT.md`와 DB 모델 갱신 |
| 관리자 ESG PDF | 산출물 형식, 다운로드 API, 저장 정책 미정 | 디자인/백엔드 구현 계획서 작성 |
| SSO/SAML | 인증 공급자, 도메인 검증, 계정 연결 정책 미정 | 인증 API 문서 확장 |
| 맞춤 도메인 자동화 | DNS/인프라/테넌트 라우팅 정책 미정 | 아키텍처 문서 확장 |

## 12. 완료 기준

- 초기 제품 범위가 가입 코드 + 회사 이메일 도메인 기반으로 정리되어 있다.
- 후속 관리자 화면, ESG 리포트, B2B 구독 결제, 회사 부담 정산의 포함/제외 범위가 분리되어 있다.
- API, DB, 아키텍처, 디자인 문서와의 교차 참조가 명시되어 있다.
- 문서화되지 않은 API/DB/디자인 동작은 `BLOCKED` 조건으로 분리되어 있다.
