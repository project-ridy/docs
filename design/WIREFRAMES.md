# Ridy — 와이어프레임

## 공통 시각 패턴

- 와이어프레임의 구조는 유지하되 실제 구현은 `DESIGN_SYSTEM.md`의 KT Seamless Flow 참고 기반 token과 컴포넌트 규칙을 따릅니다.
- 화면의 대부분은 Gray-first hierarchy로 구성하고, Accent/Feedback 색상은 CTA·상태·포커스에만 사용합니다.
- 상단 제목/뒤로가기/주요 액션은 TopAppBar 패턴으로 구현합니다.
- 하단 탭은 활성 탭에 Primary 색상과 `aria-current="page"`를 함께 적용합니다.
- 카드형 리스트는 `title → route/time → price/seats/status` 정보 순서를 유지합니다.
- `OPEN`, `MATCHED`, `PENDING`, `FAILED`, `CANCELLED` 같은 상태는 Badge + 텍스트로 표시합니다.
- Primary CTA는 각 화면의 핵심 진행 액션 하나에만 사용합니다.

## 2. 홈·검색 (탑승자 — 동네 기반 지도 탐색)

```
┌─────────────────┐
│ [성수동 → 본사 🔍] │
│  오늘 08:00 · 3km │
│─────────────────│
│                 │
│      지도        │
│   ●       ●     │
│        ●        │
│                 │
│─────────────────│
│  주변 회사행 카풀 │
│ ┌─────────────┐ │
│ │ 성수동 인근  │ │
│ │ 08:30 · 2석 │ │
│ │ 박준서 ⭐4.8 │ │
│ └─────────────┘ │
│─────────────────│
│  🏠  🕓  💬  👤 │
└─────────────────┘
```

**UI 스펙**
- 상단은 로고/알림/프로필 바가 아니라 검색 입력 역할을 합니다.
- 검색 입력 선택 시 현재 화면 위에 검색 sheet를 열고, 적용 후 같은 지도와 카드 rail을 갱신합니다.
- 하단 탭은 `홈`, `기록`, `채팅`, `프로필` 순서입니다. `검색` 탭은 별도 메뉴로 두지 않습니다.
- `기록` 탭은 지난/예정 운행 기록 또는 정산 기록 진입점이며, 검색 결과 화면으로 이동하지 않습니다.

```
│  주변 5명의 동료  │
│─────────────────│
│      지도        │
│   ●       ●     │
│        ●        │
│─────────────────│
│ ┌─────────────┐ │
│ │🚗 박준서    │ │
│ │성수동 인근   │ │
│ │08:30 · 2석  │ │
│ └─────────────┘ │
```

**UI 스펙**
- `/matchings`는 공유 링크·기존 딥링크 호환용 경로입니다.
- 새로운 독립 검색 리스트 화면을 만들지 않고 S04 홈·검색 UI를 재사용합니다.
- query string 검색 조건이 있으면 상단 검색 입력과 결과를 해당 조건으로 초기화합니다.

```
┌─────────────────┐
│  ← 차주 모드  + │
│  오늘 운행      │
│─────────────────│
│  운행 등록      │
│  [지도에서 출발 위치]│
│  표시명: 성수동 인근 │
│  목적지: 본사      │
│  [시간] [좌석]  │
│  [요금]         │
│  [운행 등록]    │
│─────────────────│
│  요청 대기 2건  │
│  ┌─────────────┐│
│  │ 김하나      ││
│  │ 강남역 픽업 ││

│  └─────────────┘│
│─────────────────│
│  🏠  🕓  💬  👤 │
└─────────────────┘
```

**UI 스펙**
- 운행 등록 영역은 지도 기반 `departure`, `pickupLabel`, `workplace`, `departureTime`, `availableSeats`, `fare` 입력을 포함합니다.
- 승인 전 탑승자에게 정확한 집 주소를 노출하지 않고 `pickupLabel`과 대략 위치만 표시합니다.
- 요청 카드에는 탑승자 이름, 픽업 위치, 요청 메시지, 수락/거절 CTA를 표시합니다.
- `OPEN`, `MATCHED`, `IN_PROGRESS`, `CANCELLED` 상태는 badge 색상으로 구분합니다.

```

│ 박준서 ⭐4.8     │
│ 개발팀 · 42회    │
│─────────────────│

│      ↓ 50분     │
│ 본사 09:20      │
│─────────────────│
│ 요금 5,000원    │
│ 잔여 좌석 2석   │
│ 흡연 불가 · 짐 OK│
```

**UI 스펙**
- 같은 기업 구성원 정보만 표시하며 외부 사용자 노출은 금지합니다.
- 승인 전 출발 위치는 표시용 동네명과 지도 marker만 노출합니다.
- 이미 요청한 운행은 Primary CTA 대신 요청 상태 badge를 표시합니다.
- 만석/취소 상태는 CTA를 disabled 처리하고 사유 문구를 표시합니다.

## 반응형 와이어프레임 규칙

| 화면 유형 | Mobile (0~639px) | Tablet (640~1023px) | Desktop (1024px+) |
|---|---|---|---|
| 인증/온보딩 | 중앙 단일 카드, 16px padding | 최대 480px 카드 중앙 정렬 | 최대 520px 카드 중앙 정렬 |
| 홈/검색 | 지도 위 상단 검색, 하단 탭 고정 | 검색 sheet + 카드 2열 허용 | 검색/필터 좌측, 지도/결과 우측 2열 |
| 상세 | 단일 컬럼, CTA 하단 sticky | 상세 카드 + 보조 정보 분리 | 상세/프로필/요청 패널 2~3열 |
| 채팅 | 메시지 전체 폭, 입력 하단 고정 | 채팅 목록/방 전환 유지 | 좌측 채팅 목록 + 우측 채팅방 |
| 정산/관리 | 요약 카드 세로 스택 | 요약 카드 2열 | 요약 카드 3~4열 + 표 레이아웃 |

공통으로 Mobile 터치 타깃은 최소 44px, 입력 높이는 48px, 하단 fixed 영역은 safe-area inset을 반영합니다.
